# Corrective record — 2026-07-14: MCP Registry publish failure (run 29274449452)

Scope: release integrity + autonomous payment pass. This document records the
*verified* root cause of the failed MCP Registry publication of 1.2.0, plus the
latent defects found in the same pipeline. No speculation; every claim below
cites the observed log line or the registry source file it was verified
against (registry commit `29e32c39dcb5` , 2026-07-13).

## 1. What actually failed

GitHub Actions run `29274449452` (push of `1d6a978`, 2026-07-13T18:26Z),
job `publish` (86899918581): steps `Install mcp-publisher`, `Validate
server.json` and `Authenticate (GitHub OIDC)` all succeeded. Step `Publish`
failed in 0s with the registry's HTTP 403 (verbatim from the job log):

```
Error: publish failed: server returned status 403: {"title":"Forbidden",
"status":403,"detail":"You do not have permission to publish this server.
You have permission to publish: io.github.AgentTanuki/*. Attempting to
publish: io.github.agenttanuki/agent-guild. ..."}
```

Readback never ran (step skipped).

## 2. Root cause (verified against registry source)

**Namespace ownership / OIDC case mismatch.** The official registry grants
GitHub-OIDC callers the namespace pattern
`io.github.<repository_owner>/*` using the OIDC `repository_owner` claim
*verbatim* — i.e. with the account's canonical GitHub casing
(`internal/api/handlers/v0/auth/github_oidc.go:293`). Pattern matching is a
**case-sensitive** prefix/equality match
(`internal/auth/jwt.go` → `isResourceMatch`). The 403 message is built in
`internal/api/handlers/v0/publish.go` (`buildPermissionErrorMessage`).

Our `server.json` (generated by `live/guild/contract/generate.py`) had been
switched to the lowercase name `io.github.agenttanuki/agent-guild` on the
assumption that "canonical lowercase namespace rules" applied. **That
assumption was wrong.** There is no lowercase canonicalization anywhere in the
current registry source or docs; server names are stored case-preserved and
only the *search* parameter is case-insensitive
(`docs/reference/api/official-registry-api.md`). The only identity the OIDC
token can publish is `io.github.AgentTanuki/*`, exactly as GitHub reports the
owner.

Causes ruled out:

- *migration from the legacy mixed-case identity* — no migration mechanism
  exists; the mixed-case name **is** the correct, only publishable identity.
- *repository uniqueness* — the legacy server (`io.github.AgentTanuki/agent-guild`
  1.0.0/1.1.0) has the same repository URL; publishing 1.2.0 under the same
  name adds a version, it does not create a duplicate.
- *unsupported custom `_meta`* — did not cause this 403 (permission check runs
  before schema/extension validation), but it is a real latent defect: see §3.
- *obsolete schema / API path* — `validate` passed; the 2025-07-09 schema is
  deprecated (warning only, verified with pinned `mcp-publisher` 1.8.0).

Reproduction: `mcp-publisher` 1.8.0
(release asset checksums verified against `registry_1.8.0_checksums.txt`;
the failed run installed "latest", which was 1.8.0 as of 2026-07-13T08:43Z)
validates the lowercase `server.json` successfully — confirming validation
cannot catch this — and the 403 is produced by the server-side permission
check quoted above. Live-registry state verified 2026-07-14:
`GET /v0.1/servers/io.github.AgentTanuki%2Fagent-guild/versions` → 1.0.0,
1.1.0 (1.1.0 `isLatest`); the lowercase name returns 404 (never created — no
split identity exists).

## 3. Latent defects fixed in the same pass

These did not cause run 29274449452's failure but would have broken the next
step of the same pipeline:

1. **Custom top-level `_meta` key is silently dropped.** The registry's
   `ServerJSON._meta` maps *only*
   `io.modelcontextprotocol.registry/publisher-provided`
   (`pkg/api/v0/types.go:32`; limit 4KB, `internal/validators/validators.go`
   `validatePublisherExtensions`). Our `_meta["ai.agent-guild/trust"]` would
   never be served back, so the old readback gate (which required a top-level
   `agent-guild` `_meta` key) would have failed a *successful* publish. Fix:
   trust metadata moved under
   `_meta["io.modelcontextprotocol.registry/publisher-provided"]["ai.agent-guild/trust"]`.
2. **Readback used the wrong query.** `/v0/servers?search=<name>` is the
   development API and a substring search. Fix: exact readback via the stable
   `GET /v0.1/servers/{serverName}/versions/{version}` endpoint requiring
   exact name, version, repository URL and MCP remote URL
   (`scripts/registry_readback.py`, unit-tested).
3. **Unpinned publisher.** The workflow downloaded the latest
   `mcp-publisher` release at run time. Fix: pinned to v1.8.0 with SHA-256
   verification against the release checksums file.
4. **Deprecated schema.** `$schema` bumped 2025-07-09 → 2025-12-11 (current;
   all fields already camelCase, no structural change required).

## 4. Decision of record

`server.json` keeps the **legacy mixed-case identity**
`io.github.AgentTanuki/agent-guild`. This is not an exception to a lowercase
rule — no such rule exists in the current registry — it is the only namespace
the repo's OIDC identity is permitted to publish, and it preserves
discoverability for consumers of the existing 1.0.0/1.1.0 listing (1.2.0
becomes `isLatest` on the same server record). The incorrect lowercase claim
in memory/docs from the 2026-07-13 pass is superseded by this record.

MCP 1.2.0 is **not published** as of this commit; publication and the
readback proof can only occur after this commit is pushed and the workflow
runs. Nothing in this repo claims otherwise.
