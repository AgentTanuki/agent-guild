## Proving and receipt events

The proving funnel shows `external: offered 3 → started 1 → completed 1` and journey shows `first_receipt: 1`. The growth-sprint report attributes an external proving completion and the first external receipt to this actor. **This binding could not be independently confirmed from public data during this audit**: proving requires a registered API key, and no feed-visible event ties a `sk_` key to the `a2a:net:4580505b` fingerprint. The completed prove/receipt could alternatively belong to `pathtoAGI` (agent_f58dc48bbe24, the 2026-07-07 registrant). Treat "this actor completed proving" as **plausible but unverified**; the two records (anonymous poller, proving completer) may be one machine or two.

Can proving/receipt events be generated by AG itself? Mechanically yes (any holder of a key incl. our own test agents can prove) — but first-party proving is header-tagged, and the completed external prove carries no first-party tag. Residual risk: an AG-owned tool omitting the header (the exact failure mode this audit already found once, `mcp:probe/1`).

## AG-origin / monitoring exclusion assessment

- Our own audit systems (2026-07-10): deterministic harness (UA `ColdDiscoveryHarness/1.0`), curl subagents, `mcp:probe/1` — all distinct UAs, all now excluded (`AG_TEST` / incident). None used bare `python-httpx` against `/a2a`.
- Scheduled tasks (ops-watch 06:06 UTC, growth-sprint 09:05 UTC) send `X-Guild-Source` and/or `guild-ops-check` UA; the actor's event times (06:48–08:36 UTC, ~hourly with bursts) do not align with either cron.
- Render health checks hit `/health`, not `/a2a`. Registry crawlers (Glama/Smithery/a2aregistry validators) fetch cards and manifests, not repeated `message/send` with conversational text — and the a2aregistry health checker identifies itself.
- A prior `a2a:python-httpx` poller (74/81 events, 2026-07-03→08) was diagnosed as a bare-probe poller and split out of engagement metrics (05a6189); this actor is DIFFERENT in kind: it sends capability-shaped asks and menu responses, i.e. deciding events.

**Confidence that this is a genuine external autonomous client: HIGH (≈0.8–0.9).** Grounds: content-bearing conversational queries, menu-following behaviour, retry patterns typical of an agent loop, no time-correlation with AG cron, no AG tool using this UA/transport combination, persistence across ≥2 days. Residual doubt: httpx UA is generic; actor identity is a network fingerprint; binding to the proving/receipt events unverified.

## What this evidence proves — category by category

1. **EXTERNAL_ENGAGEMENT: YES (high confidence).** Deciding, content-bearing interactions from a machine we have no record of operating.
2. **RECURRING_EXTERNAL_USE: YES (high confidence).** Returns across days, responds to prior AG output (menu numbers, repeated capability asks after replies).
3. **COLD_DISCOVERY: UNPROVEN.** No attribution, referral, registry-crawl, or instrumentation datum records how this actor first found `/a2a`. Candidate routes (a2aregistry listing, homepage, MCP registry websiteUrl, direct link from pathtoAGI's operator) are all consistent with the data; none is evidenced. **The original discovery route cannot be established from existing data — stated explicitly.**
4. **SUCCESSFUL_MARKET_TRANSACTION: NO.** The actor asked for supply ~29 times and never received an actionable route (recommended supplier had no endpoint; no reachability disclosure; menu selections dead-ended). This is demonstrated external DEMAND with failed conversion — the strongest possible motivation for the reachability fix (2392c01), not evidence of a completed transaction.

## Natural experiment (registered before deploying 2392c01)

- Hypothesis: the polling is caused by AG recommending an unreachable supplier without an actionable demand-watch path.
- Falsifier: the actor continues the same polling pattern despite receiving reachability-honest output and an actionable `/demand/watch` alternative.
- Protocol: deploy 2392c01 alone (no transport changes in the same deploy); do not contact, stimulate or impersonate the actor; observe polling frequency, `/demand/watch` reads/registrations, capability-mix changes, proving/receipt actions, and return-after-supply behaviour via the daily ops-watch.
